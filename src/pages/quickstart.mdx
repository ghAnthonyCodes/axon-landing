Myelin is a local tool. You write simulation blocks in Python, Myelin compiles them to C, and validates each one against the Python reference before you move on. Once your blocks are validated locally, they are ready to run at scale on AXON.

This guide walks through the full local workflow block by block.

---

## Installation

### Prerequisites

- Python 3.11+
- A C compiler — `gcc` or `clang`, already present on macOS and most Linux installs
- An Anthropic API key — Myelin calls Claude to translate your ODE and controller to C

### Clone and install

Clone Myelin somewhere on your machine:

```bash
git clone https://github.com/runaxon/myelin-engine
```

Set up a venv inside your project and install Myelin into it with the `-e` flag. This creates a live link to the Myelin source so updates are picked up automatically.

```bash
cd my-simulation
python3 -m venv .venv
source .venv/bin/activate
pip install setuptools
pip install -e ../myelin-engine
pip install anthropic python-dotenv
```

Create a `.env` file in your project root:

```
ANTHROPIC_API_KEY=sk-ant-...
```

### Project structure

Myelin works on any directory. The only requirement is a file called `myelin_adapters.py` in your project root containing your block subclasses.

```
my-simulation/
  .venv/               ← venv (myelin + your deps)
  .env                 ← ANTHROPIC_API_KEY
  my_model.py          ← your existing ODE
  myelin_adapters.py   ← Myelin blocks go here
```

## Project setup

The example used throughout this guide is a [Lotka-Volterra predator-prey system](https://github.com/runaxon/axon-plant-predator-prey) — a two-state ODE with prey population `x`, predator population `y`, and a harvest rate control input. The same steps apply to any ODE-based simulation.

## Plant

The `Plant` block defines your ODE. Myelin reads `state_size`, `input_size`, and `params_size` to size the C kernel, then translates `derivatives()` to C.

```python
# myelin_adapters.py
from myelin import Plant

class MyPlant(Plant):
    state_size  = 2   # x, y
    input_size  = 1   # harvest rate u(t)
    params_size = 4   # alpha, beta, delta, gamma

    def derivatives(self, state, inputs, params):
        x, y = state
        u_t, = inputs
        alpha, beta, delta, gamma = params
        dx = alpha * x - beta * x * y - u_t
        dy = delta * x * y - gamma * y
        return (dx, dy)
```

Compile and validate:

```bash
python ../myelin-engine/main.py --target . --adapter plant
```

Expected output:


## Measurement

The `Measurement` block defines what the controller sees. It takes the full plant state and returns a scalar. Keeping this separate from the plant means you can swap sensors without touching the ODE.

```python
# myelin_adapters.py
from myelin import Measurement

class MyMeasurement(Measurement):
    def measure(self, state, params):
        return state[0]   # controller sees prey population
```

## Controller

The `Controller` block wraps your control law. All tunable parameters are defined in `__init__` with defaults — the Optimizer will sweep them. Do not include `dt` in `__init__`: it is a scenario-level setting and is injected by the harness via `configure(dt)` before the simulation starts.

Myelin follows one level of delegation, so your controller can wrap an inner class.

```python
# myelin_adapters.py
from myelin import Controller
from controller.pid import PIDController

class MyController(Controller):
    def __init__(self, kp=1.0, ki=0.2, kd=0.4,
                 setpoint=40.0,
                 max_rate=10.0, min_rate=0.0):
        self._pid = PIDController(
            kp=kp, ki=ki, kd=kd,
            setpoint=setpoint,
            max_rate=max_rate, min_rate=min_rate,
        )

    def configure(self, dt):
        self._pid.dt = dt

    def step(self, measurement):
        return self._pid.step(measurement)

    def reset(self):
        self._pid.reset()
```

`configure(dt)` is called once before any `step()` calls, with the `dt` from the active scenario. Myelin follows the delegation into `PIDController.step()`, verifies it is pure arithmetic, and generates a C struct with `make_state()`, `step()`, and `reset()`.

## Simulator

The Simulator has no subclass to write. Once Plant, Measurement, and Controller are compiled, run:

```bash
python ../myelin-engine/main.py --target . --adapter simulator
```

Myelin reads the metadata written by each prior build step and wires all three blocks into a single C translation unit — zero Python boundary crossings per simulation step.

Expected output:

```
── Myelin ───────────────────────────────────────────────────────
  Simulator            (manifest-driven)
```

## ScenarioLoader

The `ScenarioLoader` defines the simulation settings for a single run. Myelin calls `load(scenario_id)` at runtime to get `dt`, `duration`, `target`, and any other settings for that scenario.

```python
# myelin_adapters.py
from myelin import ScenarioLoader
import json

class MyScenarioLoader(ScenarioLoader):
    default_id = 'default'   # Myelin uses this to validate load()

    def load(self, scenario_id):
        with open(f'scenarios/{scenario_id}.json') as f:
            s = json.load(f)
        return {
            'dt':       s['dt'],
            'duration': s['duration'],
            'target':   s['target'],
            'state0':   tuple(s['state0']),  # initial conditions for this scenario
        }
```

Validate:

```bash
python ../myelin-engine/main.py --target . --adapter scenario_loader
```

Expected output:

```
── Myelin ───────────────────────────────────────────────────────
```

## PatientLoader

The `PatientLoader` loads a single patient's model parameters given a patient ID. Initial conditions (`state0`) belong in the scenario — they describe where the simulation starts, not how a patient's body works.

```python
# myelin_adapters.py
from myelin import PatientLoader
import json

class MyPatientLoader(PatientLoader):
    default_id = 0   # Myelin uses this to validate load()

    def load(self, patient_id):
        with open(f'cohort/{patient_id}.json') as f:
            p = json.load(f)
        return (p['alpha'], p['beta'], p['delta'], p['gamma'])
```

Myelin validates that `params` matches `params_size` declared in your Plant.

Validate:

```bash
python ../myelin-engine/main.py --target . --adapter patient_loader
```

## Evaluator

The `Evaluator` defines what "good" means for a simulation. It has two methods:

- `score(measurements, scenario)` — computes a scalar loss for one simulation run given the full measurement vector. Myelin transpiles this to C so scoring happens inside the C kernel with no Python overhead.
- `aggregate(losses)` — combines per-patient scalars into a single job-level loss. Called once per job in Python.

```python
# myelin_adapters.py
from myelin import Evaluator

class MyEvaluator(Evaluator):
    def score(self, measurements, scenario):
        target = scenario['target']
        dt = scenario['dt']
        total = 0.0
        for m in measurements:
            total += (m - target) ** 2
        return total * dt

    def aggregate(self, losses):
        return sum(losses) / len(losses)
```

`target` comes from the scenario at runtime — set it in your `ScenarioLoader`. `aggregate()` receives one scalar per patient and returns the job-level score.

Compile and validate:

```bash
python ../myelin-engine/main.py --target . --adapter evaluator
```

## Running and Optimizing

There is no Optimizer subclass to write. The job JSON describes everything — which scenario, which patients, and which controller parameters to sweep. Scalar values are fixed. Objects with `min`, `max`, and `step` are swept.

```json
{
    "scenario_id": "100_time_units",
    "patient_ids": [1, 2, 3],
    "controller": {
        "kp": {"min": 0.0, "max": 5.0, "step": 0.1},
        "ki": {"min": 0.0, "max": 1.0, "step": 0.05},
        "kd": 0.0,
        "setpoint": 0.667,
        "max_rate": 2.0,
        "min_rate": 0.0
    },
    "evaluator": {
        "target": 0.667
    }
}
```

### Optimize

Sweep all parameter combinations across the cohort, find the best, and save to `.myelin/best_params.json`:

```bash
python ../myelin-engine/main.py --target . --optimize jobs/job.json
```

```

  [optimize] workers=14
  [optimize] running...

  [optimize] best params: {'kp': 0.0, 'ki': 0.22, 'kd': 0.0}

  [optimize] best params saved — run: --run jobs/job.json
```

### Run

Run the best params from the last `--optimize` across the cohort and save per-patient traces:

```bash
python ../myelin-engine/main.py --target . --run jobs/job.json --save-traces traces/
```

```
── Myelin ───────────────────────────────────────────────────────
  [run] scenario=100_time_units  patients=3
  [run] controller={'kp': 0.0, 'ki': 0.22, 'kd': 0.0, 'setpoint': 0.667, ...}
  [run] running...
  [run] aggregate loss: 0.047569
  [run] 3 patients in 0.002s  (1832 sims/sec)
  [run] traces written to traces/

── Done ─────────────────────────────────────────────────────────
```

Each trace is a CSV with `measurement` and `action` columns — one row per timestep.

### Debug

Override specific params inline to inspect a single patient:

```bash
python ../myelin-engine/main.py --target . --run jobs/job.json \
  --params '{"kp": 0.5, "ki": 0.1, "kd": 0.0}' --save-traces traces/
```

`--params` skips the `best_params.json` lookup and uses the inline values instead. All other fixed params still come from the job JSON.
